# Назначение документа

Документ определяет архитектурные принципы разработки GoStudy.

Любое изменение архитектуры сначала обсуждается, затем фиксируется в этом документе и только после этого реализуется в коде.

---

# Главные принципы проекта

1. Простота поддержки.
2. Максимальная модульность.
3. Отсутствие дублирования кода.
4. Масштабируемость.
5. Предсказуемая структура проекта.
6. Документация является частью проекта.

---

# Стандарт разработки

Каждая новая сущность создаётся по одинаковой схеме.

```
Обсуждение

↓

Архитектурное решение

↓

База данных

↓

REST API

↓

Frontend Service

↓

React Hook

↓

Business Components

↓

Page Components

↓

Тестирование

↓

Документация
```

Перепрыгивать этапы запрещено.

---

# Архитектура проекта

Проект разделён на независимые уровни.

```
Frontend

↓

Hooks

↓

Services

↓

REST API

↓

Database
```

Каждый уровень отвечает только за собственную область.

---

# UI Library

Все универсальные компоненты располагаются

```
src/admin/components/ui/
```

Например

* Button
* Input
* Select
* Badge
* Loader
* Pagination
* Table
* Modal
* Toolbar
* EmptyState

UI-компоненты не содержат бизнес-логики.

---

# Business Components

Располагаются

```
src/admin/components/business/
```

Например

* UserCard
* TeacherSubjects
* StudentProfileCard
* LessonStatus
* HomeworkStatus

Используются повторно в различных разделах.

---

# Структура разделов

Каждый раздел имеет собственную папку.

Например

```
Accounts/

Students/

Teachers/

Lessons/

Payments/
```

Каждый раздел полностью независим.

---

# REST API

Каждая сущность имеет собственный каталог.

```
api/admin/

users/

students/

teachers/

lessons/

payments/
```

Запрещается смешивать API различных сущностей.

---

# Жизненный цикл разработки

Новый раздел проходит одинаковые стадии.

```
Архитектура

↓

База данных

↓

API

↓

Hooks

↓

UI

↓

Тестирование

↓

Документация

```

Только после этого начинается следующий раздел.

---

# Пользователи

Раздел Accounts является техническим реестром пользователей.

Содержит

* ФИО
* Email
* Телефон
* Роль
* Статус
* Дату регистрации
* Последний вход

Не содержит

* анкету ученика;
* анкету преподавателя;
* учебные данные;
* документы.

---

# Ученики

Профиль ученика располагается отдельно.

Раздел отвечает за

* анкету;
* родителей;
* преподавателей;
* уроки;
* домашние задания;
* оплаты.

---

# Преподаватели

Профиль преподавателя располагается отдельно.

Раздел отвечает за

* анкету;
* предметы;
* документы;
* проверку;
* учеников;
* отзывы.

---

# Жизненный цикл аккаунта

Допустимые статусы

```
active

blocked

deleted
```

где

active

Аккаунт работает.

blocked

Вход запрещён.

deleted

Архив.

Физическое удаление пользователей запрещено.

---

# Архив пользователей

Архивирование не удаляет информацию.

Сохраняются

* профиль;
* уроки;
* домашние задания;
* сообщения;
* отзывы;
* платежи;
* история действий.

Аккаунт может быть восстановлен.

---


# Ответственность администратора

Администратор управляет системой.

Не занимается редактированием персональных данных пользователей.

Допустимые действия

* просмотр;
* смена роли;
* блокировка;
* разблокировка;
* архивирование;
* восстановление;
* переход в профиль.

---

# Ответственность модератора

Модератор обладает ограниченными правами.

Полный перечень определяется отдельно.

---

# Правила разработки

Каждый компонент должен выполнять только одну задачу.

Большие компоненты разделяются.

Новые возможности должны использовать существующую архитектуру.

Временные решения ("костыли") запрещены.

---

Документация является частью проекта.

Любое архитектурное изменение фиксируется в документации.

# Архитектурные решения (ADR)

## ADR-001

**Дата:** 28.06.2026

### Users содержит только учетные записи.

Анкеты учеников и преподавателей вынесены в отдельные разделы.

**Статус:** Принято.

---

## ADR-002

**Дата:** 28.06.2026

### Физическое удаление пользователей запрещено.

Используется жизненный цикл:

```
active → blocked → deleted (архив)
```

История пользователя сохраняется полностью.

**Статус:** Принято.

---

## ADR-003

**Дата:** 28.06.2026

### UI Library отделена от Business Components.

UI содержит универсальные компоненты.

Business Components содержат бизнес-логику проекта.

**Статус:** Принято.

---

## ADR-004

**Дата:** 28.06.2026

### Каждый новый раздел создаётся по единому стандарту.

```
Архитектура

↓

База данных

↓

API

↓

Services

↓

Hooks

↓

UI

↓

Тестирование

↓

Документация
```

**Статус:** Принято.

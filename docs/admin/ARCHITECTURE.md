# Основные принципы

GoStudy строится как долгосрочная образовательная платформа.

Приоритеты разработки:

1. Простота поддержки.
2. Высокая модульность.
3. Минимум дублирования кода.
4. Максимальная читаемость.
5. Расширяемость без переписывания существующих модулей.

---

# Принципы разработки

## Модульность

Каждый компонент отвечает только за одну задачу.

Нельзя создавать большие универсальные файлы.

Если компонент начинает выполнять несколько функций — он разделяется.

---

## UI библиотека

Все универсальные компоненты располагаются в

```text
src/admin/components/ui/
```

Например

* Button
* Input
* Select
* Badge
* Loader
* EmptyState
* Pagination
* Table
* Toolbar
* Modal

UI-компоненты ничего не знают о бизнес-логике проекта.

---

## Business Components

Компоненты, относящиеся к GoStudy:

```text
src/admin/components/business/
```

Например

* UserCard
* UserStatusBadge
* TeacherSubjects
* StudentInfo
* LessonStatus
* HomeworkStatus

Используются повторно в различных разделах.

---

## Структура страниц

Каждый раздел имеет собственную папку.

Пример

```text
pages/

    Accounts/

    Students/

    Teachers/

    Lessons/

    Payments/
```

Внутри располагаются только компоненты данного раздела.

---

# API

Каждая сущность имеет собственный каталог.

Пример

```text
api/admin/

    users/

    students/

    teachers/

    lessons/

    payments/
```

API разных сущностей не смешиваются.

---

# Users

Раздел "Аккаунты"

Является исключительно реестром пользователей.

Users НЕ содержит анкеты преподавателя или ученика.

Карточка пользователя показывает только:

* ФИО
* Email
* Телефон
* Роль
* Статус
* Дату регистрации
* Последний вход

---

# Student Profile

Анкета ученика располагается отдельно.

```
students/profile
```

---

# Teacher Profile

Анкета преподавателя располагается отдельно.

```
teachers/profile
```

---

# Жизненный цикл аккаунта

Допустимые статусы

```
active
blocked
deleted
```

где

active

Аккаунт активен.

blocked

Вход запрещён.

deleted

Архив.

Физическое удаление пользователей запрещено.

---

# Архив пользователей

При архивации сохраняется полностью вся история пользователя.

Не удаляются:

* уроки
* домашние задания
* сообщения
* отзывы
* платежи
* журнал действий

Пользователь может быть восстановлен в любой момент.

---

# Поля архива

Таблица users содержит

```
archived_at

archived_by

archive_reason
```

---

# Роли

Допустимые роли

```
student

teacher

moderator

admin
```

---

# Права администратора

Администратор может

* менять роль
* блокировать
* разблокировать
* архивировать
* восстанавливать
* просматривать карточки
* переходить в профиль

Администратор НЕ редактирует персональные данные пользователя.

---

# Права модератора

Определяются отдельно.

По умолчанию модератор имеет меньше прав, чем администратор.

---

# Правила разработки

Перед созданием нового компонента необходимо определить его тип.

UI

или

Business

или

Page Component

---

# Принцип завершения разделов

Новый раздел начинается только после полного завершения предыдущего.

Раздел считается завершённым если

* API полностью готово
* Интерфейс полностью готов
* Код соответствует архитектуре
* Основные сценарии протестированы

---
